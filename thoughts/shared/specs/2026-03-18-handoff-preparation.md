## Решение по хостингу: Timeweb Cloud VPS

### Почему именно VPS, а не shared и не Vercel

```
Три варианта — и почему выбрали VPS:

❌ Timeweb Shared (текущий, 634 руб/мес)
   - Не умеет запускать Node.js/Next.js
   - Подходит только для PHP-сайтов (WordPress)
   - Аналогия: как пытаться вставить кассету VHS в DVD-плеер

❌ Vercel (бесплатно)
   - Может быть заблокирован РКН в России
   - Для сайта госкомитета — недопустимый риск
   - Посетители из РФ могут не зайти на сайт

✅ Timeweb Cloud VPS (~477 руб/мес)
   - Российский сервер — 100% доступен в РФ
   - Поддерживает Node.js/Next.js
   - Полный контроль — файлы, настройки, бэкапы
   - Дешевле текущего shared (477 < 634 руб!)
   - Клиент уже в Timeweb — просто меняет тариф
```

### Как это будет работать

```
[Посетитель] → вводит opora-migration.ru →
  → [DNS Timeweb] направляет на VPS-сервер →
    → [Nginx на VPS] передаёт запрос в Next.js →
      → [Next.js] показывает сайт

Простая аналогия:
- Домен (opora-migration.ru) — адрес на вывеске магазина
- DNS — указатель "магазин вот там →"
- Nginx — охранник на входе (проверяет и направляет)
- Next.js — сам магазин с товарами (сайт)
```

### Что нужно от клиента

1. **Купить VPS** в панели Timeweb Cloud (вместо shared или в дополнение)
   - Минимальный тариф: 1 vCPU, 1 GB RAM, 15 GB SSD (~477 руб/мес)
   - Или: перейти с shared на VPS (можно сделать прямо в панели Timeweb)
2. **Предоставить доступ** к панели Timeweb (логин/пароль от аккаунта ck68396)
3. **Или:** дать SSH-доступ к VPS после создания (IP, логин, пароль)

#### Задача 1.2: Rate limiting (ограничение частоты запросов)

**Идеальный результат:** С одного IP-адреса можно отправить максимум 5 форм в минуту. Если больше — сервер отвечает "подождите".

**Что это простым языком:** Как турникет в метро — пропускает по одному. Если кто-то пытается пролезть 100 раз подряд — блокируем на минуту.

**Шаги:**
1. Создать файл `src/lib/rate-limit.ts` с простым счётчиком по IP
2. Используем Map в памяти (на VPS процесс один — Map работает стабильно)
3. Подключить в `src/app/api/send-email/route.ts`
4. При превышении лимита возвращать HTTP 429 "Слишком много запросов"

#### Задача 1.3: Honeypot-защита от ботов

**Идеальный результат:** Боты-спамеры не могут отправить форму. Люди не замечают никакой разницы.

**Что это простым языком:** Мы добавляем в форму невидимое поле. Человек его не видит и не заполняет. А бот-робот заполняет все поля подряд — и попадается. Как ловушка для мышей — мышь лезет на сыр, а человек мимо проходит.

**Шаги:**
1. Добавить скрытое поле `<input name="website" style="display:none">` в формы
2. На сервере проверять: если поле `website` заполнено — это бот, отклоняем
3. Обновить оба компонента форм

**Файлы:**

- `src/app/contacts/page.tsx` — добавить honeypot-поле

**Что это простым языком:** Как личный кабинет в интернет-банке, только для сайта. Заходишь по адресу opora-migration.ru/admin, вводишь логин и пароль — и можешь всё менять. Никаких технических знаний не нужно.

#### Задача 3.1: Авторизация админки

**Идеальный результат:** Только человек с правильным логином и паролем может войти в админку. Пароль хранится в настройках сервера (.env), а не в коде.

**Как это работает (простым языком):**
1. Клиент открывает opora-migration.ru/admin
2. Видит форму: логин + пароль
3. Вводит пароль → сервер проверяет → выдаёт "пропуск" (токен)
4. "Пропуск" действует 24 часа — потом нужно войти снова
5. Без "пропуска" ни одна страница админки не откроется

**Что такое JWT-токен:** Это как электронный пропуск в бизнес-центр. Ты показываешь паспорт на входе (логин/пароль), тебе выдают карточку (токен). Карточка открывает все двери в здании. Через 24 часа карточка перестаёт работать — нужно снова показать паспорт.

**Шаги:**
1. Установить пакет `jose` — библиотека для работы с токенами
2. Создать `src/lib/auth.ts` — генерация и проверка JWT-токена
3. Создать `src/app/api/admin/login/route.ts` — проверка логина/пароля
4. Создать `src/middleware.ts` — "охранник", который проверяет токен на всех страницах /admin

#### Задача 3.2: API для управления данными

**Идеальный результат:** Набор "команд" для админки — получить данные, сохранить, удалить. Все защищены токеном (без "пропуска" не работают).

**API-маршруты:**
```
POST   /api/admin/login          — вход (логин + пароль → токен)
GET    /api/admin/contacts       — получить текущие контакты
PUT    /api/admin/contacts       — сохранить изменённые контакты
GET    /api/admin/news           — список всех новостей
POST   /api/admin/news           — добавить новую новость
PUT    /api/admin/news           — обновить существующую новость
DELETE /api/admin/news           — удалить новость
GET    /api/admin/texts          — получить тексты страниц
PUT    /api/admin/texts          — сохранить изменённые тексты
POST   /api/admin/upload         — загрузить фото
GET    /api/admin/images         — список загруженных фото
DELETE /api/admin/images         — удалить фото
```

**Шаги:**
1. Создать каждый API-маршрут в папке `src/app/api/admin/`
2. Каждый маршрут проверяет JWT-токен перед выполнением
3. Данные читаются/записываются в JSON-файлы (через функции из `src/lib/data.ts`)
4. Фото сохраняются в папку `public/uploads/`

**Идеальный результат:** Простой и понятный интерфейс на русском языке. Клиент видит разделы, нажимает — редактирует — сохраняет. Как в WordPress, только проще.

### ЭТАП 6: Подготовка деплоя на VPS (30 мин)

**Цель:** Подготовить всё для запуска сайта на Timeweb Cloud VPS.

**Что это простым языком:** Мы готовим "коробку с инструкцией" — чтобы сайт можно было установить на VPS за 10 минут. Как конструктор IKEA — открыл, собрал по инструкции, работает.

#### Задача 6.1: Dockerfile

**Что это:** Рецепт для автоматической установки сайта на сервер. Вместо 20 команд вручную — одна команда `docker compose up`.

**Что такое Docker простым языком:** Это как контейнер для перевозки. Мы кладём сайт + все его зависимости в "контейнер". Этот контейнер можно поставить на любой сервер — и всё будет работать одинаково. Не нужно устанавливать Node.js, настраивать версии — всё уже внутри контейнера.

**Файлы:**
- `Dockerfile` — рецепт сборки контейнера
- `docker-compose.yml` — настройки запуска
- `.dockerignore` — что не включать в контейнер

#### Задача 6.2: Конфигурация Nginx

**Что это:** Nginx — это "охранник-привратник" на сервере. Он принимает запросы из интернета и направляет их в нужное место. Также выдаёт HTTPS-сертификат (замочек в браузере).

**Файл:** `deploy/nginx.conf`

**Что делает:**
- Принимает запросы на opora-migration.ru
- Перенаправляет в Next.js (порт 3000)
- Отдаёт статику (картинки, CSS) напрямую — быстрее
- Настраивает HTTPS через Let's Encrypt (бесплатный сертификат)

#### Задача 6.3: Инструкция по деплою

**Файл:** `docs/DEPLOY.md`

**Содержание (пошагово):**
1. Купить VPS на Timeweb Cloud (минимальный тариф)
2. Подключиться по SSH
3. Установить Docker и Docker Compose (2 команды)
4. Склонировать проект с GitHub

### Почему Timeweb VPS — идеальный вариант для 152-ФЗ

**Закон 152-ФЗ** говорит: персональные данные граждан РФ при первичном сборе должны храниться на серверах в России.

**Наша ситуация с Timeweb VPS:**
1. ✅ **Сервер в России** — Timeweb дата-центры в Москве и Санкт-Петербурге
2. ✅ **Формы** (имя, email, телефон) → отправляются через SMTP на **mail.ru** (тоже Россия)
3. ✅ **Данные на сервере** — JSON-файлы хранятся на российском VPS
4. ✅ **Политика конфиденциальности** — есть, со ссылкой на 152-ФЗ
5. ✅ **Согласие на обработку** — чекбоксы на формах

**Вывод:** Полное соответствие 152-ФЗ. Все данные хранятся и обрабатываются на территории РФ.
